# Strategic Brief

> 매주 일요일 밤 한 번씩 다시 읽어라.
> 결정에 막히면 이 문서로 돌아와라.
> 이 문서가 흔들리면 회사가 흔들린다.

---

## 0. 단 한 줄

> **"우리는 가성불량 제거기를 만드는 게 아니라, SMT 검사 데이터의 표준 layer를 만든다. 2차 검사기는 그 첫 wedge일 뿐이다."**

이 한 줄이 무너지면 다 무너진다.

---

## 1. 세 개의 렌즈 — 매 결정마다 동시에 봐라

### 1.1 빌 게이츠 렌즈 — 플랫폼 > 제품
> *"다른 사람이 우리 위에서 빌드해줄 수 있나?"*

- 표준을 정의한 자가 권력을 갖는다 (Bluetooth는 노키아 것이 아니다)
- 어댑터 SDK를 외부에 공개하지 않으면 그건 SaaS이지 플랫폼이 아니다
- 12-24개월 안에 vendor-side 엔지니어가 자기 어댑터 직접 짜는 그림이 나와야 함
- 개발자 문서 = 영업 문서 (Stripe·Twilio가 보여준 길)

### 1.2 워렌 버핏 렌즈 — 해자 > 시장
> *"18개월 뒤에도 우리만 갖고 있는 것은 무엇인가?"*

- 모델은 카피된다. **데이터·스키마·고객 신뢰는 카피 안 된다**
- 가격은 비용이 아니라 ROI에서 나온다 — ₩5M = "무시해도 됨"의 시그널
- Switching cost는 시간에 비례해 커지도록 설계해야 함 (§5 별도 섹션)
- 이해 못하는 사업에 시간 쓰지 말 것 — 우리가 이해하는 것: SMT 검사 워크플로우

### 1.3 일론 머스크 렌즈 — First Principle > 관습
> *"왜 가성불량이 30%인가?"*

- AOI 벤더가 sensitivity > specificity로 튜닝하니까 (escape 1건 책임 vs FC 100건 무시)
- → 우리는 그 비대칭을 흡수하는 **인프라 레이어** = 벤더의 적이 아니라 **보완재**
- 90일 production 1건 > 365일 perfect 솔루션
- 매뉴팩처링 그 자체가 제품이다 — 공장 운영을 더 좋게 만드는 것이 진짜 가치

## 2. 매일 결정 직전 5초 필터

결정 누르기 전에 머릿속으로 통과시켜라:

```
1. 데이터 플라이휠을 빠르게 하나?           (버핏)
2. 표준화 쪽인가, 커스텀 쪽인가?            (게이츠)
3. 90일 내 production에 넣을 수 있나?       (머스크)
4. 18개월 후에도 우리만 가능한가?           (버핏)
5. ₩100M 회사로 끝나나, ₩10B로 가나?        (셋 다)
```

5개 중 3개 이상 "yes"면 진행. 아니면 멈추고 다시 설계.

## 3. 안티패턴 — 절대 하지 마라

| 함정 | 왜 위험 |
|------|---------|
| 모델 더 정교화 | 이미 충분하다. 첫 production이 더 급함 |
| LOI 0건에 가격 협상 | leverage 0인 협상은 항상 진다 |
| DGX Spark 자본 지출 | 첫 LOI 받기 전에는 Colab + 기존 GPU로 충분 |
| 코드 99% 완성도 추구 | production 1건이 코드 1만 줄보다 무겁다 |
| 단일 anchor 의존 | 협상 종료 = 회사 종료. 항상 3개 동시 |
| 모든 어댑터 직접 작성 | 외부 개발자가 짜야 플랫폼이 된다 |
| 고객 데이터 클라우드 export | 온프레 약속을 첫날 깨지 말 것 |
| 경쟁사 가격 따라가기 | 우리는 ROI 가격 — 경쟁사는 비용 가격 |

## 4. 자본·시간 배분

시간 100을 어디에 쓰나:

| 항목 | 현재 (추정) | 권장 | 변화 |
|------|------------|------|------|
| 모델 정교화 | 50 | 15 | ↓↓ |
| 어댑터 SDK·표준화 | 20 | 30 | ↑ |
| **영업·고객 확보** | **5** | **35** | **↑↑↑ ⭐** |
| 데이터 플라이휠 | 5 | 15 | ↑ |
| 기타 (HW·UI·문서) | 20 | 5 | ↓ |

**가장 큰 시정**: 영업 5 → 35. 모델은 충분하다. 부족한 것은 첫 logo다.

## 5. Switching Cost — 해자의 실체

> **"고객이 우리 솔루션에서 다른 솔루션으로 바꿀 때 드는 모든 고통의 총량."**
> 시간에 비례해 커지도록 의식적으로 설계해야 함.

### 5.1 시간 곡선

```
바꾸는 비용 (= switching cost)

  높음 │                                    ╱── 3년차: "이걸 빼면 라인 멈춤"
       │                              ╱──
       │                         ╱──
       │                    ╱──
       │              ╱──
       │        ╱── 1년차: 큰 결단 필요
       │   ╱──
       │ ╱
  낮음 │╱  1주차: 그냥 박스 빼면 됨
       └────────────────────────────────►
                  사용 기간
```

**좋은 사업은 이 곡선이 가파르게 올라가도록 의식적으로 설계**한다. 평평하면 누가 더 싼 솔루션 내면 다음 분기에 교체된다.

### 5.2 우리 사업의 두 가지 메커니즘

#### 🧠 라벨 누적 (Label Accumulation)

운영자가 매일 PASS/DEFECT 클릭하는 행위 자체가 라벨이다.

| 기간 | 누적 라벨 | 의미 |
|------|----------|------|
| 1주차 | ~5,000건 | 모델이 일반적 베이스라인 그대로 |
| 6개월 | ~150,000건 | 그 라인 특유의 false-call 패턴 학습 |
| 3년차 | ~1,000,000건 | 모델이 그 고객 라인에 완전히 특화 — 다른 누구도 못 만드는 모델 |

**고객이 떠나려고 하면**: 라벨 export는 가능하지만, 그걸 받아서 즉시 우리 수준 모델을 만들 수 있는 회사가 없음 → 가성불량률이 5% → 30%로 회귀 → 운영자 야근, 라인 처리량 ↓, 부서장 KPI 박살. 결론: **"지금 5%인데 30%로 돌아가는 위험 못 감당"** → 떠날 수 없음.

#### 🕸️ 오케스트레이션 깊이 (Orchestration Depth)

처음엔 작게 박혀있다가 점점 더 깊이 침투한다.

```
Phase 1 (0-12개월) : 2차 검사기 옆 어시스턴트
                     떼는 비용: 박스 1개 빼면 끝

Phase 2 (12-24개월): SPI/ICT/functional test 데이터 흡수, 라인 traceability
                     떼는 비용: 다른 시스템도 연동 끊어야 함

Phase 3 (24-36개월): MES augmentation, 수율 KPI 산출 책임
                     떼는 비용: 운영 SW 일부 다시 짜야 함, 라인 정지

Phase 4 (36+)      : 생산 routing 결정 일부 담당
                     떼는 비용: 거의 회사 시스템 갈아엎기
```

단순 가성불량 분류기로만 머물면 Phase 1에서 끝난다. **인접 워크플로우를 한 단계씩 흡수**하도록 설계해야 깊이가 생긴다.

### 5.3 의식적으로 해야 할 설계 결정

| 결정 | Switching cost ↑ 방향 (해야 할 것) | Switching cost ↓ 방향 (피해야 할 것) |
|------|-----------------------------------|--------------------------------------|
| 라벨 데이터 | 우리 SQLite에 누적, export는 가능하나 우리 모델 없이는 의미 약함 | 표준 포맷 즉시 export, 어디서나 그대로 사용 가능 |
| MES 통합 | 우리만의 augmented data가 MES로 흐름 (sidecar) | 단순 verdict pass-through |
| KPI 대시보드 | 운영자·관리자 일과 화면이 **우리 UI** | 단순 이메일 리포트 |
| 어댑터 범위 | 점진적으로 SPI/ICT/functional test까지 흡수 | AOI 한 종류만 |
| 학습 모델 | 그 라인 데이터로 점점 특화되는 사이트 모델 | 모든 고객에 동일한 글로벌 모델 |

오른쪽 컬럼대로 만들면 = 평범한 SaaS. 왼쪽 컬럼대로 만들면 = Salesforce급 lock-in.

### 5.4 비유

- **AWS**: 1주차 = EC2 서버 1대 빌림 (떠나기 쉬움). 5년차 = PB 데이터 + Lambda + DynamoDB + IAM + VPC → 옮기는 데만 1년 + 수십억. 5년 이상 고객 이탈률 1% 미만.
- **Salesforce**: 1주차 = CRM 1라이선스. 5년차 = 영업·마케팅·고객지원·파트너·BI + 영업사원 머슬 메모리 + 50개 커스텀 앱 → 교체 = 영업 6개월 정지.

**우리는 이걸 SMT 검사·제조에서 해야 한다.**

## 6. 이번 주 4가지 베팅

```
[1] aoi-common-spec 공개      → 표준 권력 선점
[2] ₩0 무료 파일럿 프로그램   → anchor 락인
[3] 의사결정자 3명 미팅       → LOI 시드
[4] DGX Spark 주문 보류       → 자본 보존
```

이 순서대로. (1)+(2)는 코드/문서 작업. (3)+(4)는 사업 결정.
